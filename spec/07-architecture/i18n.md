# Internationalization (i18n)

> Version: 1.0 | Last updated: 2026-03-20

This document defines the internationalization architecture using next-intl with cookie-based locale detection, message file structure, component usage patterns, and date/number formatting.

---

## 1. Overview

| Aspect | Decision |
|---|---|
| Library | next-intl |
| Strategy | Cookie-based locale, **no URL prefix** |
| Primary locale | `ja` (Japanese) |
| Secondary locale | `en` (English) |
| Locale cookie | `NEXT_LOCALE` |
| Detection order | Cookie → `Accept-Language` header → fallback to `ja` |
| Content scope | **UI strings only** — user-generated content stays in Japanese |
| Backend i18n | None — API returns all text in Japanese |

---

## 2. Locale Detection Flow

```
Request arrives
    │
    ▼
Check NEXT_LOCALE cookie
    │
    ├── Cookie found: "en" → use English
    ├── Cookie found: "ja" → use Japanese
    │
    └── Cookie not found
            │
            ▼
        Check Accept-Language header
            │
            ├── Contains "en" → use English
            └── Default → use Japanese (ja)
```

### 2.1 Configuration

```typescript
// i18n/config.ts
export const locales = ["ja", "en"] as const;
export type Locale = (typeof locales)[number];
export const defaultLocale: Locale = "ja";
```

```typescript
// i18n/request.ts
import { getRequestConfig } from "next-intl/server";
import { cookies, headers } from "next/headers";
import { defaultLocale, locales, type Locale } from "./config";

export default getRequestConfig(async () => {
  const cookieStore = await cookies();
  const headerStore = await headers();

  // Priority 1: Cookie
  const cookieLocale = cookieStore.get("NEXT_LOCALE")?.value;
  if (cookieLocale && locales.includes(cookieLocale as Locale)) {
    const messages = (await import(`./messages/${cookieLocale}.json`)).default;
    return { locale: cookieLocale, messages };
  }

  // Priority 2: Accept-Language header
  const acceptLanguage = headerStore.get("accept-language") || "";
  const preferredLocale = acceptLanguage.includes("en") ? "en" : defaultLocale;

  const messages = (await import(`./messages/${preferredLocale}.json`)).default;
  return { locale: preferredLocale, messages };
});
```

## 3. Locale Switching

### 3.1 UI Toggle

Locale switching is provided in two locations:
- **Header**: Icon toggle button (globe icon)
- **Footer**: Language selector link

### 3.2 Implementation

```typescript
// components/shared/locale-toggle.tsx
"use client";

import { useLocale } from "next-intl";
import { useRouter } from "next/navigation";
import { Globe } from "lucide-react";
import { Button } from "@/components/ui/button";

export function LocaleToggle() {
  const locale = useLocale();
  const router = useRouter();

  const toggleLocale = () => {
    const nextLocale = locale === "ja" ? "en" : "ja";

    // Set cookie (expires in 1 year)
    document.cookie = `NEXT_LOCALE=${nextLocale}; path=/; max-age=${60 * 60 * 24 * 365}; SameSite=Lax`;

    // Reload to apply new locale (server needs to re-render with new messages)
    router.refresh();
  };

  return (
    <Button variant="ghost" size="icon" onClick={toggleLocale} aria-label="Toggle language">
      <Globe className="h-4 w-4" />
      <span className="ml-1 text-xs">{locale === "ja" ? "EN" : "JA"}</span>
    </Button>
  );
}
```

### 4.3 Translation Key Naming Convention

| Pattern | Example | Description |
|---|---|---|
| `{namespace}.{section}.{key}` | `events.detail.host` | Nested namespace with dot-separated keys |
| `{namespace}.{action}` | `profile.save` | Direct action under namespace |
| `{namespace}.empty` | `events.empty` | Empty state message for a section |
| `common.actions.{verb}` | `common.actions.save` | Shared action buttons |
| `errors.{category}.{key}` | `errors.notFound.title` | Error messages |

**Rules:**
- Keys are always in English, camelCase
- Use descriptive names reflecting UI context, not technical identifiers
- Shared strings go in `common` namespace
- Feature-specific strings go in their feature namespace
- Error messages go in `errors` namespace

### 5.3 Dynamic Values (ICU Format)

```typescript
// Using interpolation
t("common.pagination.page", { page: 3, total: 10 })
// → "Page 3 of 10"

t("common.footer.copyright", { year: 2026 })
// → "© 2026 VRChat October Cohort Community"

t("clubs.gallery.count", { count: 12 })
// → "12 images"
```

### 5.4 Plural Rules (English)

next-intl supports ICU message format for plurals:

```json
{
  "clubs": {
    "gallery": {
      "count": "{count, plural, =0 {No images} one {1 image} other {# images}}"
    }
  }
}
```

Japanese does not have plural rules, so the `ja.json` file uses a simpler form:

```json
{
  "clubs": {
    "gallery": {
      "count": "{count}枚の画像"
    }
  }
}
```

## 6. Date/Time Formatting

### 6.1 date-fns with Locale

```typescript
// lib/date.ts
import { format, formatDistanceToNow, parseISO } from "date-fns";
import { ja } from "date-fns/locale/ja";
import { enUS } from "date-fns/locale/en-US";
import type { Locale } from "@/i18n/config";

const DATE_LOCALES = {
  ja: ja,
  en: enUS,
} as const;

export function formatDate(dateStr: string, locale: Locale): string {
  const date = parseISO(dateStr);
  return format(date, locale === "ja" ? "yyyy年M月d日" : "MMM d, yyyy", {
    locale: DATE_LOCALES[locale],
  });
}

export function formatDateTime(dateStr: string, locale: Locale): string {
  const date = parseISO(dateStr);
  return format(date, locale === "ja" ? "yyyy年M月d日 HH:mm" : "MMM d, yyyy HH:mm", {
    locale: DATE_LOCALES[locale],
  });
}

export function formatRelativeTime(dateStr: string, locale: Locale): string {
  const date = parseISO(dateStr);
  return formatDistanceToNow(date, {
    addSuffix: true,
    locale: DATE_LOCALES[locale],
  });
}
```

### 6.2 Format Examples

| Input | Japanese (ja) | English (en) |
|---|---|---|
| `2026-03-15T19:00:00Z` (date) | `2026年3月15日` | `Mar 15, 2026` |
| `2026-03-15T19:00:00Z` (datetime) | `2026年3月15日 19:00` | `Mar 15, 2026 19:00` |
| `2026-03-15T19:00:00Z` (relative) | `5日後` | `in 5 days` |

### 6.3 Usage in Components

```typescript
// Server Component
import { getLocale } from "next-intl/server";
import { formatDate } from "@/lib/date";

export default async function EventDetail({ event }: { event: PublicEventDetail }) {
  const locale = await getLocale();
  return <time>{formatDate(event.start_time, locale as Locale)}</time>;
}

// Client Component
import { useLocale } from "next-intl";
import { formatDate } from "@/lib/date";

export function EventCard({ event }: { event: PublicEvent }) {
  const locale = useLocale();
  return <time>{formatDate(event.start_time, locale as Locale)}</time>;
}
```

## 7. Number Formatting

Use the native `Intl.NumberFormat` API:

```typescript
// lib/date.ts (or a separate number-format utility)
import type { Locale } from "@/i18n/config";

export function formatNumber(value: number, locale: Locale): string {
  return new Intl.NumberFormat(locale === "ja" ? "ja-JP" : "en-US").format(value);
}
```

| Value | Japanese | English |
|---|---|---|
| `1234` | `1,234` | `1,234` |
| `1000000` | `1,000,000` | `1,000,000` |

## 8. Root Layout Integration

```typescript
// app/layout.tsx
import { NextIntlClientProvider } from "next-intl";
import { getLocale, getMessages } from "next-intl/server";

export default async function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  const locale = await getLocale();
  const messages = await getMessages();

  return (
    <html lang={locale}>
      <body>
        <NextIntlClientProvider locale={locale} messages={messages}>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  );
}
```

## 9. SEO Implications

| Aspect | Impact |
|---|---|
| URL structure | **No locale prefix** — `/events` not `/en/events` |
| Indexing | Only the default locale (Japanese) is indexed by search engines |
| Hreflang | Not applicable — cookie-based locale produces the same URL |
| `<html lang>` | Set dynamically based on cookie: `<html lang="ja">` or `<html lang="en">` |
| OG meta content | Always in Japanese (primary audience) |

> **Limitation**: Since locale is determined by a cookie and not the URL, search engines (which don't send cookies) always receive the Japanese version. This is acceptable because the primary audience is Japanese-speaking VRChat community members.

## 10. Content Translation Scope

| Content Type | Translated? | Source |
|---|---|---|
| Navigation labels | Yes | `i18n/messages/` |
| Button text | Yes | `i18n/messages/` |
| Form labels & placeholders | Yes | `i18n/messages/` |
| Error messages (UI) | Yes | `i18n/messages/` |
| Empty state messages | Yes | `i18n/messages/` |
| Page titles & descriptions | Yes | `i18n/messages/` + `generateMetadata` |
| Event titles/descriptions | **No** | API (Japanese only) |
| Member bios | **No** | API (user-authored, Japanese) |
| Club names/descriptions | **No** | API (Japanese only) |
| Tags | **No** | API (Japanese only) |
| Date/time display | Yes | `date-fns` with locale |

## 11. Adding a New Translation Key

1. Define the key in both `ja.json` and `en.json`
2. Use the appropriate namespace prefix
3. In Server Components: `const t = await getTranslations("namespace")`
4. In Client Components: `const t = useTranslations("namespace")`
5. Access: `t("key")` or `t("nested.key")`
6. With variables: `t("key", { variable: value })`
