# トレードオフ

> **対象読者**: アーキテクト、コントリビューター
>
> **ナビゲーション**: [ドキュメントホーム](../README.md) > [設計](README.md) > トレードオフ

## 概要

すべての設計決定にはトレードオフが伴います。VRC Web-Backend で行った主要なトレードオフ、各選択の理由、再検討する条件を記録します。

---

## 1. Go/TypeScript ではなく Rust

| 側面 | 詳細 |
|-----|------|
| **決定** | バックエンド言語・フレームワークとして Axum を備えた Rust を使用 |
| **検討した代替案** | Go (Gin/Echo)、TypeScript (Express/Fastify)、Python (FastAPI) |

### メリット

- コンパイル時安全性: 所有権、ライフタイム、型システムがバグの全クラスを防止
- 教育的価値: 「ロマンを追求する厳格さ」哲学に整合
- パフォーマンス: ゼロコスト抽象化、GC なし、予測可能なレイテンシ

### デメリット

- コントリビューターにとって急な学習曲線
- 長いコンパイル時間（フルビルドで約2-3分）
- Node.js/Go より小さなエコシステム

### 再検討する条件

- 最小限のオンボーディング時間で多くのコントリビューターが必要な場合
- コンパイル時間が深刻なボトルネックになった場合

---

## 2. 単純なレイヤードではなくヘキサゴナルアーキテクチャ

| 側面 | 詳細 |
|-----|------|
| **決定** | 明示的なポートトレイトとアダプター実装によるヘキサゴナルアーキテクチャ |
| **検討した代替案** | 単純な3層（handler → service → repository）、MVC、バーティカルスライス |

### メリット / デメリット

| メリット | デメリット |
|---------|----------|
| 純粋でテスト可能なドメインコア | より多くのボイラープレート（トレイト + 実装） |
| 交換可能なインフラ | より多くのファイルと間接参照 |
| 明確な依存方向 | 単純な CRUD 操作にはやり過ぎ |

---

## 3. ランタイムチェックではなく型状態認可

| 側面 | 詳細 |
|-----|------|
| **決定** | コンパイル時ロール強制のため `AuthenticatedUser<R: Role>` にファントム型パラメータ |
| **検討した代替案** | ハンドラ内ランタイムロールチェック、ミドルウェアベース RBAC、属性ベースアクセス制御 |

### メリット / デメリット

| メリット | デメリット |
|---------|----------|
| 不正アクセスがコンパイルエラー | 関数シグネチャの複雑なジェネリック境界 |
| ロールチェック忘れがゼロ | 動的権限変更に対して柔軟性が低い |

---

## 4. ORM ではなく SQLx コンパイル時検証

| 側面 | 詳細 |
|-----|------|
| **決定** | コンパイル時クエリ検証とオフラインモードを備えた SQLx を使用 |
| **検討した代替案** | Diesel (Rust ORM)、SeaORM、検証なし生 SQL 文字列 |

### メリット / デメリット

| メリット | デメリット |
|---------|----------|
| SQL エラーがビルド時に検出 | 開発中に PostgreSQL の実行が必要 |
| クエリの完全な制御 | クエリ変更時に手動で `sqlx-data.json` の更新が必要 |
| 型安全な結果マッピング | ORM ベースアプローチより手動 SQL が多い |

---

## 5. 分散ではなくシングルノードデプロイメント

| 側面 | 詳細 |
|-----|------|
| **決定** | Docker Compose による単一 Proxmox VM（app + PostgreSQL + Caddy） |
| **検討した代替案** | Kubernetes、Docker Swarm、ロードバランサー付きマルチノード、サーバーレス |

### メリット / デメリット

| メリット | デメリット |
|---------|----------|
| 単純な運用 | 単一障害点 |
| 低コスト | 垂直スケーリングのみ |
| 高速デプロイ | アップグレード中のダウンタイム |

### 再検討する条件

- コミュニティが約1,000同時ユーザーを超えた場合
- 稼働率 SLA が 99.9% 以上の可用性を要求する場合

---

## 6. Email/パスワードではなく Discord 専用認証

| 側面 | 詳細 |
|-----|------|
| **決定** | 唯一の認証方法として Discord OAuth2 |
| **検討した代替案** | Email/パスワード + Discord 連携オプション、複数 OAuth2 プロバイダー |

### メリット / デメリット

| メリット | デメリット |
|---------|----------|
| パスワード管理ゼロ | Discord の稼働率への強い依存 |
| 自然なコミュニティ検証 | Discord を持たないユーザーの排除 |
| 攻撃面の縮小 | フォールバック認証方法なし |

---

## 7. JWT ではなくサーバーサイドセッション

| 側面 | 詳細 |
|-----|------|
| **決定** | PostgreSQL に SHA-256 ハッシュトークンを保存するサーバーサイドセッション |
| **検討した代替案** | JWT（ステートレス）、リフレッシュトークン付き JWT、Redis バックセッション |

### メリット / デメリット

| メリット | デメリット |
|---------|----------|
| 即座の失効 | リクエストごとのデータベースルックアップ |
| 小さなトークンサイズ | 水平スケーリングに共有 DB が必要 |
| 完全なサーバーサイド制御 | 期限切れセッションの定期クリーンアップが必要 |
