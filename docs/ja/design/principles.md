# 設計原則

> **対象読者**: 全員
>
> **ナビゲーション**: [ドキュメントホーム](../README.md) > [設計](README.md) > 原則

## 概要

これらの原則は VRC Web-Backend のすべての設計・実装決定を導きます。優先度順にリストされており、原則が矛盾する場合は上位の原則が優先されます。

---

## 1. 「ロマンを追求する厳格さ（Romance Through Rigor）」

**学習と正しさを最大化する場合、最も困難な実行可能なパスを選択する。**

このプロジェクトは本番ソフトウェアと教育の交差点にあります。Rust、ヘキサゴナルアーキテクチャ、または形式検証を選択するのは、それらが簡単だからではなく、正しく実装する苦闘が深い理解と堅牢なソフトウェアを生み出すからです。

「実行可能」がキーワードです。パスは動作する、保守可能なソフトウェアにつながる必要があります。正しさや学習に寄与しない複雑さは拒否されます。

**実践:**
- Go/TypeScript ではなく Rust — 借用チェッカーが所有権とライフタイムの推論を教える
- ランタイムロールチェックではなく型状態認可 — ファントム型が高度な型システムの使用法を教える
- 「trust me」コメントではなく Kani 証明 — 形式検証がコードに関する数学的推論を教える

---

## 2. ランタイムチェックより型安全性

**型システムがバグを防げるなら、必ず防ぐ。**

ランタイムチェックは最後の手段です。目標はコンパイル時に無効な状態を表現不可能にすることです。`AuthenticatedUser<Admin>` 型の値を見れば、ランタイム状態を確認することなく、そのリクエストが検証済みの管理者からのものであることがわかります。

**実践:**
- `AuthenticatedUser<R: Role>` — ファントム型によるロール強制
- `ValidatedJson<T>` — ハンドラがデータを見る前に入力バリデーション実行
- `NonEmpty<T>`、有界整数 — 型にエンコードされたドメイン不変条件

---

## 3. 可能な限りコンパイル時検証

**エラー検出を可能な限り左にプッシュする — 理想的には `cargo build` まで。**

コンパイル時に捕捉されたエラーは、本番環境に到達し得ないエラーです。Rust コンパイラ、SQLx のオフラインモード、カスタム derive マクロを使用して、バイナリ生成前に正しさを検証します。

**実践:**
- SQLx コンパイル時クエリ検証 — SQL 構文エラー、型の不一致、欠落カラムがビルド中に検出
- `#[derive(Validate)]` — フィールドバリデーションルールがマクロ展開時にチェック
- `#[derive(ErrorCode)]` — エラーコード導出がすべてのバリアントにコードがあることを保証

---

## 4. 多層防御

**単一のセキュリティメカニズムを単独で信頼しない。すべてのレイヤーは前のレイヤーが失敗したと想定する。**

セキュリティは重複するレイヤーで実装されます。XSS 防止はサニタイズだけに依存せず、サニタイズ後の検出チェックを追加します。セッショントークンは単に比較されるだけでなく、まずハッシュ化され、その後定時間で比較されます。

**実践:**
- XSS 防止（3層）:
  1. `pulldown-cmark` による Markdown レンダリング（制御された HTML 生成）
  2. `ammonia` による HTML サニタイズ（許可リストベースのタグ/属性フィルタリング）
  3. 多層防御の `<script>` 検出（サニタイザーバイパスを捕捉）

- トークン比較（2層）:
  1. SHA-256 ハッシュ化（可変長比較からのタイミングリーク防止）
  2. `subtle::ConstantTimeEq`（バイト単位比較からのタイミングリーク防止）

---

## 5. ヘキサゴナル純粋性

**ドメインコアは外部依存ゼロ。HTTP、SQL、Discord について一切知らない。**

ドメイン型とビジネスロジックは標準ライブラリのみに依存する純粋な Rust コアに存在します。データベース、HTTP フレームワーク、サードパーティ API などすべての外部関心事は、ドメインが定義するが実装しないポートトレイトを通じてアクセスされます。

---

## 6. エラー網羅性

**キャッチオールのエラーハンドリングなし。すべてのエラーバリアントが明示的にマッピングされる。**

代数的エラー型（Rust enum）とレイヤー間の明示的な `From` 変換を使用します。`anyhow::Error` なし、`Box<dyn Error>` なし、エラーマッピングのキャッチオール `_` マッチアームなし。すべてのドメインエラーが特定の API エラーにマッピングされ、特定の HTTP レスポンスにマッピングされます。

---

## 7. プロパティベース正しさ

**重要な不変条件は例示ベーステストだけでなく、proptest と Kani で検証される。**

例示ベーステストは特定の入力が期待される出力を生成することを証明します。プロパティベーステストは入力の*クラス*が不変条件を満たすことを証明します。Kani 証明はコードの正しさに関する有界な数学的保証を提供します。

---

## 原則の相互作用

| シナリオ | 解決策 |
|---------|--------|
| 型安全性がより多くのボイラープレートを必要とする | ボイラープレートを受け入れる — 型安全性が勝つ（原則 2 > 利便性） |
| コンパイル時チェックにライブ DB が必要 | SQLx オフラインモードを妥協案として使用（原則 3 を適応） |
| 多層防御がレイテンシを追加する | 軽微なレイテンシを受け入れる — セキュリティが勝つ（原則 4 > パフォーマンス） |
| ヘキサゴナル純粋性がより多くのトレイトを意味する | トレイトを受け入れる — ドメイン純粋性が勝つ（原則 5 > 簡潔性） |
| エラー網羅性がより多くの match アームを意味する | match アームを受け入れる — 網羅性が勝つ（原則 6 > 簡潔性） |
